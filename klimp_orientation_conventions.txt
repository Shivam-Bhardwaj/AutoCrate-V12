KLIMP ORIENTATION CONVENTIONS - L-BRACKET GEOMETRY
==================================================

KLIMP PHYSICAL DESCRIPTION:
- L-shaped angle brackets with two perpendicular sides
- Long side and short side meet at 90° angle
- Width runs perpendicular to both sides (bracket thickness)
- Purpose: Structural corner reinforcement between crate panels

KLIMP STRUCTURAL FUNCTION:
- Hold front panel together with top and side panels
- Provide corner reinforcement at panel intersections
- Connect multiple panels for structural integrity

COORDINATE SYSTEM REFERENCE:
- X: Sideways (left/right)
- Y: Away from screen (depth into screen)  
- Z: Up (vertical)

KLIMP ORIENTATION BY PANEL LOCATION:

TOP PANEL KLIMPS (KL_1 through KL_10):
- Position: Mounted on top surface (Y = 0)
- Long side: Extends downward into crate (-Z direction)
- Short side: Extends away from viewer (+Y direction)
- Width: Spans sideways (±X direction)
- Purpose: Connects top panel to front/back panels

SIDE PANEL KLIMPS (Left: KL_11-20, Right: KL_21-30):
- Position: Mounted on vertical panel surfaces
- Long side: Extends away from viewer (+Y direction)
- Short side: Extends toward crate center (±X direction)
- Width: Spans vertically (±Z direction)  
- Purpose: Connects side panels to front/back panels

PROBLEM WITH EULER ANGLE APPROACH:
- Euler angles (RX, RY, RZ) suffer from gimbal lock
- Order dependency (XYZ vs ZYX rotation order matters)
- Non-intuitive for complex orientations
- Difficult to interpolate between orientations

PROPOSED SOLUTIONS:

OPTION 1: DIRECTION VECTORS (RECOMMENDED)
=========================================
Define each klimp with:
- Origin Point: KL_XX_X, KL_XX_Y, KL_XX_Z (translation)
- X-Axis Direction: KL_XX_X_DIR_X, KL_XX_X_DIR_Y, KL_XX_X_DIR_Z
- Y-Axis Direction: KL_XX_Y_DIR_X, KL_XX_Y_DIR_Y, KL_XX_Y_DIR_Z
- Z-Axis Direction: KL_XX_Z_DIR_X, KL_XX_Z_DIR_Y, KL_XX_Z_DIR_Z

Benefits:
+ No gimbal lock
+ Intuitive orientation definition
+ Easy to visualize and debug
+ Works with any CAD system
+ Direct mapping to transformation matrices

Example for KL_1:
KL_1_X = 10.000          // Origin position
KL_1_Y = 20.000
KL_1_Z = 30.000
KL_1_X_DIR_X = 1.000     // X-axis points along global X
KL_1_X_DIR_Y = 0.000
KL_1_X_DIR_Z = 0.000
KL_1_Y_DIR_X = 0.000     // Y-axis points along global Y
KL_1_Y_DIR_Y = 1.000
KL_1_Y_DIR_Z = 0.000
KL_1_Z_DIR_X = 0.000     // Z-axis points along global Z
KL_1_Z_DIR_Y = 0.000
KL_1_Z_DIR_Z = 1.000

OPTION 2: QUATERNIONS
=====================
Define orientation with quaternions:
- Origin Point: KL_XX_X, KL_XX_Y, KL_XX_Z
- Quaternion: KL_XX_Q_W, KL_XX_Q_X, KL_XX_Q_Y, KL_XX_Q_Z

Benefits:
+ No gimbal lock
+ Compact representation (4 values vs 9)
+ Smooth interpolation
+ Mathematically elegant

Drawbacks:
- Less intuitive for manual input
- Requires quaternion math knowledge
- Harder to debug visually

OPTION 3: AXIS-ANGLE REPRESENTATION
===================================
Define rotation with single axis and angle:
- Origin Point: KL_XX_X, KL_XX_Y, KL_XX_Z
- Rotation Axis: KL_XX_AXIS_X, KL_XX_AXIS_Y, KL_XX_AXIS_Z (unit vector)
- Rotation Angle: KL_XX_ANGLE (degrees around the axis)

Benefits:
+ No gimbal lock
+ Intuitive for single-axis rotations
+ Compact (4 values)

Drawbacks:
- Complex for arbitrary orientations
- Still has singularities at 0° and 180°

OPTION 4: TWO-VECTOR APPROACH (SIMPLIFIED)
==========================================
Define orientation with just two vectors:
- Origin Point: KL_XX_X, KL_XX_Y, KL_XX_Z
- Primary Direction: KL_XX_PRIMARY_X, KL_XX_PRIMARY_Y, KL_XX_PRIMARY_Z
- Secondary Direction: KL_XX_SECONDARY_X, KL_XX_SECONDARY_Y, KL_XX_SECONDARY_Z

The third axis is computed as cross product.

Benefits:
+ Only 6 orientation values needed
+ Intuitive for most applications
+ No gimbal lock

RECOMMENDED IMPLEMENTATION FOR AUTOCRATE:
=========================================

I recommend OPTION 1 (Direction Vectors) because:

1. INTUITIVE: Engineers can easily visualize orientations
2. CAD-FRIENDLY: Most CAD systems work well with direction vectors
3. DEBUGGING: Easy to verify orientations visually
4. FLEXIBLE: Can represent any orientation
5. STABLE: No mathematical singularities

PROPOSED VARIABLE STRUCTURE:
============================

For each klimp (1-30), define 12 variables:

POSITION (3 variables):
KL_XX_X                 // Origin X coordinate
KL_XX_Y                 // Origin Y coordinate  
KL_XX_Z                 // Origin Z coordinate

X-AXIS DIRECTION (3 variables):
KL_XX_X_DIR_X          // X-axis direction X component
KL_XX_X_DIR_Y          // X-axis direction Y component
KL_XX_X_DIR_Z          // X-axis direction Z component

Y-AXIS DIRECTION (3 variables):
KL_XX_Y_DIR_X          // Y-axis direction X component
KL_XX_Y_DIR_Y          // Y-axis direction Y component
KL_XX_Y_DIR_Z          // Y-axis direction Z component

Z-AXIS DIRECTION (3 variables):
KL_XX_Z_DIR_X          // Z-axis direction X component
KL_XX_Z_DIR_Y          // Z-axis direction Y component
KL_XX_Z_DIR_Z          // Z-axis direction Z component

LEGACY COMPATIBILITY:
KL_XX_SUPPRESS         // 0=hide, 1=show
KL_XX_ROTATE          // Legacy angle (deprecated)

VALIDATION RULES:
================
1. Each direction vector should be unit length (magnitude = 1.0)
2. The three direction vectors should be orthogonal (perpendicular)
3. Right-handed coordinate system: Z = X × Y

COMMON ORIENTATIONS EXAMPLES:
============================

Identity (no rotation):
X_DIR = (1, 0, 0)
Y_DIR = (0, 1, 0)  
Z_DIR = (0, 0, 1)

90° rotation about Z-axis:
X_DIR = (0, 1, 0)
Y_DIR = (-1, 0, 0)
Z_DIR = (0, 0, 1)

90° rotation about Y-axis:
X_DIR = (0, 0, -1)
Y_DIR = (0, 1, 0)
Z_DIR = (1, 0, 0)

Face toward center of crate:
X_DIR = (-1, 0, 0)  // Points inward
Y_DIR = (0, 1, 0)   // Points up
Z_DIR = (0, 0, -1)  // Points backward

MIGRATION STRATEGY:
==================
1. Keep existing RX, RY, RZ variables for backward compatibility
2. Add new direction vector variables
3. Convert existing Euler angles to direction vectors
4. Phase out Euler angles over time
5. Update NX model to use direction vectors

This approach will eliminate gimbal lock while providing intuitive, 
debuggable orientation control for all klimps.
